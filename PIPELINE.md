# stepwise-rtl-refinement — Pipeline Contract

**Version:** 1.0 | **Date:** 2026-05-02 | **Deadline:** 2026-05-30

This document is the single source of truth for all 4 pipeline stages. Every agent reads it on startup. Any schema change requires a version bump here before implementation begins.

---

## Pipeline Overview

```
NL Spec (00_nl_spec.json)
        │
        ▼
[Stage 1: Formalization]  ← Group 1
        │ 01_formal_spec.json
        ▼
[Stage 2: Refinement]     ← Group 2  ◄── function-preserving rules (Group 3)
        │ 02_pluscal_impl.json
        ▼
[Stage 3: Codegen]        ← Group 4
        │ 03_rtl_output.json
        ▼
[Stage 4: Evaluation]     ← Groups 3 + 4
        │ 04_eval_report.json
        ▼
      [Done]
```

**Orchestration:** LangGraph. Each stage is a Python node with signature `def <stage>_node(state: PipelineState) -> PipelineState`.

**State passing:** File-based only. Nodes never share in-memory state across boundaries. `PipelineState` carries only `run_id: str` and control signals (retry counts, halt flags).

---

## Artifact Store Layout

All artifacts live under `artifacts/<run_id>/`. The `run_id` is a UUID4 generated by the LangGraph entry node and passed to every stage.

```
artifacts/
  <run_id>/
    00_nl_spec.json          # pipeline input
    01_formal_spec.json      # Stage 1 output
    02_pluscal_impl.json     # Stage 2 output
    03_rtl_output.json       # Stage 3 output
    04_eval_report.json      # Stage 4 output (terminal)
    tla/                     # .tla and .cfg files (Stage 1)
    pluscal/                 # PlusCal .tla files (Stage 2)
    rtl/                     # .bsv and .v files (Stage 3)
    benchmarks/              # benchmark result artifacts (Stage 4)
```

| Stage | LLM | Formal Tools | Compilation | Lint / Sim / Synth |
|-------|-----|-------------|-------------|-------------------|
| 1 Formalization | Claude Sonnet 4.6 | `tla2sany`, `tlc` | — | — |
| 2 Refinement | — (rules are hand-authored) | `tlc` | — | — |

**Orchestration:** LangGraph (Python 3.11+). **Schema validation:** pydantic v2 at every node entry.

**Conventions:** Single clock (`clk`), synchronous active-high reset (`rst_n` active-low), 32-bit default data width.

## Stub Protocol (Week 1 — Non-Negotiable)

Every stage must have a schema-conformant hardcoded stub committed by end of Week 1. The LangGraph skeleton runs end-to-end with all stubs before any real logic is written. This decouples all 4 groups for the first 10 days.

**Universal stub design:** 2-bit synchronous counter (`design_name: counter_2bit`).

| Stage | Stub Content |
|-------|-------------|
| Stage 1 | `01_formal_spec.json` — `state_variables: [{name: count, domain: 0..3}]`, invariant `count <= 3`, liveness `<>(count = 3)` |
| Stage 2 | `02_pluscal_impl.json` — `state_variables: [{bsv_mapping: "Reg#(Bit#(2))"}]`, `processes: [{bsv_mapping: "rule Increment"}]` |
| Stage 3 | `03_rtl_output.json` + hand-written `counter.v` passing `verilator --lint-only` |
| Stage 4 | `04_eval_report.json` — plausible dummy scores, `demo_summary` filled in |

## Critical Path

**Stage 2 (Refinement) is the critical path node.** It has the highest technical risk and is serially dependent on Stage 1. Any slip in Stage 2 directly delays Stage 3 and Stage 4.

**Protection strategy:** The Stage 2 stub is the integration contract that unblocks Groups 3 and 4 from day 1. Even if real refinement logic is incomplete, downstream stages are never blocked because they use the stub.

TLC model checks are bounded to 10^6 states. If TLC times out, set `proof_status: "pending_tlc"` and proceed — this is acceptable for demo day.

## Risk Register

| Risk | Likelihood | Mitigation |
|------|-----------|-----------|
| Refinement tree search non-termination (state explosion) | High | TLC bounded to 10^6 states; `pending_tlc` status allows pipeline to continue; backtracking depth capped at 10 |
| BSV compiler rejects LLM-generated BSV | Medium | Direct structural Verilog fallback developed in parallel, validated in Week 1 |
| LLM TLA+ syntactically valid but semantically wrong | High | Semantics judge node (Week 3); human review for all demo designs in Weeks 1–2 |
| Benchmark port signatures mismatch generated Verilog | Medium | Port adapter generated from `port_list` field in `03_rtl_output.json` (Week 2) |
| `bsv_mapping` fields too underspecified for Stage 3 | Medium | Groups 2 and 4 cross-review and sign off by end of Week 1 |
